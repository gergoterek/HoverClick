## Product Goal

v1.0 should be a stable, shippable, low-surprise release. It should confirm and polish the already-shipped v0.9.0 surface instead of becoming a risky feature release.

The recommended v1.0 theme is conservative readiness:

- keep left-click and optional right-click focus behavior stable;
- keep manual `Check for Updates...`;
- keep `Automatically Check for Updates` user-controlled and default off;
- keep automatic download/install disabled;
- keep the GitHub Release DMG plus GitHub Pages appcast workflow;
- keep Hover Click Assist out of the product surface;
- keep event tap and mouse event semantics unchanged.

## Readiness Assessment

v0.9.0 appears close enough to become v1.0 after minor polish and checklist validation. No known blocker requires a runtime feature, updater behavior change, signing change, package workflow change, or appcast workflow change.

The strongest reason to do one v1.0 readiness branch before release prep is documentation and user-facing polish: the repository still contains older wording that should be reviewed for stale v0.8.0/v0.9.0 references, release-description language, and the removed Hover Click Assist placeholder before v1.0 is presented publicly.

## Blockers

Known hard blockers: none found in this planning pass.

Conditional blockers before v1.0:

- Any failed manual update-path validation from an updater-enabled build to a future release candidate.
- Any stale public-facing docs that still describe the current release as v0.8.0 or describe the removed Hover Click Assist placeholder as present.
- Any appcast, GitHub Release, or DMG mismatch for the v1.0 release candidate.
- Any regression in Accessibility onboarding, runtime revocation fail-open behavior, left-click focus, right-click focus, menu behavior, or diagnostics.
- Any accidental change to app name, bundle identifier, signing identity, Sparkle public key, event tap mask, or mouse event pass-through semantics.

- Keep v1.0 as a polish/readiness branch only unless the user explicitly reprioritizes.

## Manual Validation Checklist

Manual Finder UI validation -- not run automatically.

- Install or launch the signed app bundle normally, never the raw binary.
- Confirm the menu shows `HoverClick` and `v0.9.0` before v1.0 version prep.
- Confirm `Left Click Focus` is enabled when Accessibility is granted.
- Confirm `Right Click Focus` is off by default and works when enabled.
- Confirm background Finder, Chrome, and a terminal/editor window focus from left click.
- Confirm right-click context menus remain native with `Right Click Focus` off and on.
- Confirm menu bar, status-item, popover, and compact overlay clicks pass through.
- Confirm Finder context-menu follow-up left-click remains native.
- Confirm background text first-drag remains documented as a known limitation, not treated as fixed.
- Confirm `Copy Diagnostics Summary` reports permission, updater, event tap, focus, and known-safety fields accurately.
- Confirm `Check for Updates...` is present and behaves through Sparkle's standard UI.
- Confirm `Automatically Check for Updates` can be enabled and disabled without enabling automatic download/install.
- Confirm `About HoverClick...` shows the expected version/build/bundle ID after release-prep metadata changes.

## Docs And GitHub Polish

Plan these changes on the next implementation branch:

- README: update latest-release wording from v0.8.0 to the current v0.9.0 baseline until v1.0 release prep.
- README: remove the old `Hover > Hover Click Assist` menu row and describe `Automatically Check for Updates`.
- README: keep the safety promise clear: no mouse-move focus, no synthetic clicks, no cursor movement, no event replay.
- GitHub Release wording: draft concise v1.0 release notes focused on stable click focus, conservative updater behavior, Accessibility onboarding, diagnostics, and known limitations.
- GitHub project/repository page: ensure any short description does not imply hover focus, AutoRaise behavior, or silent updating.
- Docs: mark v0.9.0 as the latest public baseline and v1.0 as the planned stable readiness release.

## Updater Validation Plan

The v1.0 updater test should prove the current design, not expand it.

- Validate manual `Check for Updates...` from an older updater-enabled build to a future v1.0 release candidate or test appcast item.
- Validate the no-update state against the live appcast before release-candidate publication.
- Validate the update-available state only with a final or controlled test appcast that points at the intended DMG.
- Confirm `Automatically Check for Updates` persists when toggled and remains default off for fresh installs.
- Confirm enabling automatic checks does not enable automatic download/install.
- Confirm `SUAutomaticallyUpdate = false` and `SUAllowsAutomaticUpdates = false` remain unchanged.
- Confirm `scripts/prepare-appcast.sh` remains dry-run by default and does not publish.
- Confirm every appcast item uses the final GitHub Release DMG URL, correct version/build, correct length, and Sparkle signature.
- Confirm no private key material, local paths, placeholder URLs, duplicate assets, or internal DMGs enter the appcast or repository.

## Accessibility Validation Plan

Do not reset TCC automatically. Use fresh-flow validation only when the user intentionally provides a safe fresh user, VM, clean install, or manual permission state.

- Confirm first launch without Accessibility shows the native prompt and HoverClick explanatory alert.
- Confirm HoverClick does not open System Settings automatically.
- Confirm `Left Click Focus` and `Right Click Focus` are disabled while Accessibility is missing.
- Confirm `Check Again` refreshes permission state and dismisses stale onboarding UI after permission is granted.
- Confirm activation and status-menu-open refreshes update permission state without repeated prompts.
- Confirm runtime Accessibility revocation fails open: normal mouse events return unchanged and no AX focus work runs.
- Confirm permission regrant plus `Check Again` can recover without relaunch when possible.
- Confirm diagnostics report onboarding, trust prompt, permission-missing pass-through/removal, and latest permission refresh state accurately.

## Diagnostics Sanity

- Diagnostics should report automatic update checks, automatic download/install state, and automatic install allowance.
- Diagnostics should not report the removed no-op Hover Click Assist placeholder as current behavior.
- Diagnostics should preserve stable last real/background click fields even after menu/status clicks.
- Diagnostics should show the event tap mask as left mouse down plus right mouse down only.
- Diagnostics should make permission-missing fail-open and stale tap removal visible.
- Diagnostics should retain the concise safety note that normal mouse events pass through unchanged.

## Branch Cleanup Planning

Branch cleanup should be a separate, explicit maintenance task after v1.0 planning is merged. Do not delete branches automatically.

Suggested future procedure:

1. Fetch and list merged local and remote branches.
2. Classify candidates as already merged, historical release branches, active planning branches, or unknown.
3. Keep `main`, `origin/main`, release tags, `gh-pages`, and any branch with unmerged work.
4. Present deletion candidates to the user with exact branch names and merge status.
5. Delete only after explicit approval for that exact list.

Likely inspection candidates include older `design-*`, `feature-*`, and release-prep branches that have already merged into `main`, but they must be inspected before any cleanup recommendation becomes actionable.

## Deferred Features And Non-Goals

These must stay out of v1.0 unless the user explicitly reprioritizes and accepts a separate risk branch:

- real Hover Click Assist;
- Click-Time Hover Assist;
- Scroll Focus;
- Key Focus / Caps Lock Focus;
- Excluded Apps;
- Finder selection hack;
- background click-and-drag assist;
- background text drag fix;
- new mouse event types;
- event tap mask expansion beyond `kCGEventLeftMouseDown` and `kCGEventRightMouseDown`;
- mouse-moved, mouse-dragged, mouse-up, or scroll handling;
- synthetic clicks;
- event replay;
- delayed click delivery;
- cursor movement;
- `CGEventPost`;
- `CGEventCreateMouseEvent`;
- `CGDisplayMoveCursorToPoint`;
- `CGWarpMouseCursorPosition`;
- AutoRaise-style hover-to-window-focus;
- signing identity change;
- bundle identifier change;
- app name change;
- Accessibility/TCC reset;
- Developer ID/notarization work unless split into an explicit release-signing task;
- background automatic update download/install.

Menu persistence should be deferred to v1.x unless it is proven to be a tiny docs/UI polish item with no custom menu tracking loop, no focus-stealing helper window, and no runtime input behavior change. It is not needed for v1.0.

## Recommendation

Proceed toward v1.0 as a conservative polish/readiness release only. Do not use v1.0 to introduce new event/input behavior.

Recommended next implementation branch: `feature-v1.0-readiness-polish`.

Recommended Code Engine model level for that branch: `mid`.

Use `high` only if the next task touches signing, permissions, updater/release workflow, package/appcast workflow, or runtime/event/input behavior.

A fresh Strategy starter should be written after this plan is merged. It should direct the next branch to perform v1.0 readiness polish, update public-facing docs, prepare manual validation checklists, and explicitly avoid runtime, release, signing, package, appcast, and event/input changes.
