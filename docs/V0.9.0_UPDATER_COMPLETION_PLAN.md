## Theme

v0.9.0 - Complete Updater & 1.0 Readiness.

## What Complete Updater Means For v0.9.0

Complete updater for v0.9.0 should mean a conservative, user-consented update experience, not silent background replacement.

Recommendations:

- Keep `Check for Updates...` as the primary, always-visible manual path.
- Keep the GitHub Release DMG plus GitHub Pages appcast architecture.
- Keep the Sparkle private key in the Keychain under `com.gergoterek.HoverClick`; do not add private key files, signing secrets, or generated secret material to the repository.
- Keep the app name, bundle identifier, and signing identity unchanged.
- Keep background automatic download/install disabled for v0.9.0 and v1.0.
- Do not set `SUEnableAutomaticChecks = true` as an unconditional default.
- Do not rely on an unexpected second-launch Sparkle prompt for automatic checks in v0.9.0. Sparkle supports that prompt when `SUEnableAutomaticChecks` is not set, but HoverClick should avoid adding another surprise launch-time prompt while Accessibility and Launch at Login onboarding already exist.
- Add an explicit user-facing `Automatically Check for Updates` menu setting in the implementation branch if it can be wired cleanly to Sparkle's `automaticallyChecksForUpdates` API. The setting should default off, use a native checkmark, and sit next to the updater action without becoming a diagnostic row.
- Automatic checks, once enabled by the user, may periodically check the existing appcast and notify through Sparkle's normal UI. They must not install silently.
- Keep `SUAutomaticallyUpdate = false`.
- Keep `SUAllowsAutomaticUpdates = false` unless a later explicit release decision accepts automatic downloads/installs.

This gives v1.0 a clear final updater UX:

- Manual `Check for Updates...` always works.
- Optional periodic update checks are user-enabled and reversible through `Automatically Check for Updates`.
- Update availability, release notes, download, install, and relaunch use Sparkle's standard UI.
- No update silently installs in the background.
- Release metadata and update payloads remain auditable through the appcast and GitHub Release asset.

## Recommended v0.9.0 Implementation Scope

Recommended next implementation branch: `feature-v0.9.0-updater-completion`.

Implement these items:

1. Keep the existing manual `Check for Updates...` menu item.
2. Add a small `Automatically Check for Updates` menu toggle only if Sparkle's runtime setting can be read and changed safely without custom updater state.
3. Keep initial automatic checking default off unless the user explicitly opts in.
4. Keep automatic download/install disabled.
5. Confirm `scripts/verify-app.sh` verifies the intended Sparkle settings and does not accidentally bless silent update behavior.
6. Strengthen release/appcast documentation so a future v0.9.0 appcast cannot point to an internal DMG or duplicate asset.
7. Keep the existing `scripts/prepare-appcast.sh` safety model: dry-run by default, explicit `--write`, no upload, no tag, no release creation, no private key material.
8. Add test-matrix coverage for manual checks, optional automatic checks, no silent install, appcast correctness, and 1.0 readiness gates.
9. Do not change the event tap mask or click semantics.
10. Do not add Click-Time Hover Assist, real Hover Click Assist, mouse movement handling, synthetic clicks, event replay, cursor movement, delayed click delivery, scroll focus, or key focus.

## v0.9.0 Implementation Outcome

Implemented on `feature-v0.9.0-updater-completion`:

- Manual `Check for Updates...` remains available as the explicit user-initiated Sparkle action.
- `SUFeedURL` remains `https://gergoterek.github.io/HoverClick/appcast.xml`.
- `SUPublicEDKey` remains `093ZOOvjGmr8WkI31IzBnjGwM3GXZU1q/qgDgADWm9o=`.
- `SUEnableAutomaticChecks` remains `false`, so the default install state does not silently enable periodic checks or trigger Sparkle's permission prompt.
- A new top-level `Automatically Check for Updates` menu toggle lets the user opt in or out. It writes through Sparkle's `automaticallyChecksForUpdates` property, so Sparkle owns persistence in the host app defaults.
- Automatic download/install remains disabled. `SUAutomaticallyUpdate = false` and `SUAllowsAutomaticUpdates = false` remain unchanged, and the toggle path reasserts `automaticallyDownloadsUpdates = NO`.
- Copied diagnostics now report automatic update checks, automatic download/install, and whether automatic download/install is allowed.
- No appcast, DMG, tag, GitHub Release, or `gh-pages` work was done.
- No Sparkle private key material was printed, exported, stored, or committed.
- No signing identity, bundle identifier, app name, event tap mask, or mouse event semantics changed.

Do not include these in v0.9.0:

- A custom GitHub polling/downloading updater.
- Silent or mostly silent background install.
- Developer ID/notarization transition unless split into a separate release-signing task.
- Bundle ID, app name, or signing identity changes.
- Appcast publication, DMG packaging, tag creation, or release creation outside explicit release scope.

## Hover Click Assist Decision

Hover Click Assist is out of scope for v0.9.0.

Product decision:

- Do not implement real Hover Click Assist for v0.9.0.
- Do not implement Click-Time Hover Assist for v0.9.0.
- Do not implement any hover/event semantics feature for v0.9.0.
- Treat the current `Hover > Hover Click Assist` item as a no-op placeholder that should not remain in the user-facing 1.0 surface unless it is explicitly hidden or labeled as unavailable.

Recommended cleanup:

- Include removal or hiding of the Hover Click Assist placeholder inside `feature-v0.9.0-updater-completion` as a small UI/docs cleanup, because it directly supports 1.0 readiness and removes user confusion. This cleanup is implemented in the branch.
- Keep the cleanup narrow: remove or hide the menu item, remove no-op diagnostics and log wording, and update docs/tests that currently promise the placeholder.
- Preserve historical release notes that describe what v0.8.0 shipped.
- Do not add a replacement feature, no-op replacement toggle, hover tracking, mouse movement event tap, synthetic click, event replay, cursor movement, or delayed assist path.

Docs/diagnostics references that would need cleanup in the implementation branch:

- `HoverClick.mm` menu setup, persisted `hoverClickAssistEnabled` handling, no-op logs, copied diagnostics fields, and placeholder status text.
- `docs/CURRENT_STATE.md` current product behavior and diagnostics summary.
- `docs/ARCHITECTURE.md` experimental placeholder sections.
- `docs/TEST_MATRIX.md` rows that expect the `Hover` submenu and `Hover Click Assist` toggle.
- `docs/WORKFLOW.md` manual checklist rows that still ask the tester to confirm the placeholder.

If the updater implementation becomes larger than expected, split the cleanup into `feature-v0.9.0-remove-hover-assist-placeholder`. The default recommendation is to batch it with `feature-v0.9.0-updater-completion` because it is a non-semantic UI cleanup and avoids shipping a confusing v1.0 menu surface.

Outcome: the placeholder is removed on `feature-v0.9.0-updater-completion`; no separate removal branch is needed for this cleanup.

## 1.0 Readiness Gates

Before v1.0, all of these must be true:

- Updating works from v0.8.0 or v0.9.0 to the next release through Sparkle's manual `Check for Updates...` path.
- Optional automatic checks can be enabled and disabled through `Automatically Check for Updates` and do not silently install updates.
- The appcast workflow is repeatable from a real release DMG and final GitHub Release URL.
- The appcast never leaks private key material, local paths, placeholders, or secret words.
- GitHub Release asset naming remains stable: `HoverClick-<version>.dmg`.
- The public appcast points to the intended GitHub Release DMG, not an internal DMG.
- The release process produces no duplicate public/internal DMG assets for the same tag.
- `scripts/prepare-appcast.sh` remains non-publishing by default.
- `scripts/package-dmg.sh` is not treated as the public appcast publishing workflow.
- The update validation checklist exists and includes both no-update and update-available cases.
- First-install Accessibility onboarding remains stable.
- Runtime Accessibility revocation still fails open and does not block normal clicks.
- Left Click Focus still works.
- Right Click Focus still works when enabled.
- Diagnostics are accurate and not misleading.
- No user-facing no-op Hover Assist feature remains unless explicitly hidden or clearly labeled unavailable.
- The app name, bundle identifier, signing identity, and Sparkle public key remain intentionally unchanged.
