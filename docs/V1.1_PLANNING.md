# HoverClick v1.1.x Planning

## Purpose

This document defines the next HoverClick planning cycle after the stable `v1.0.0` / build `39` release.

This is a docs-only planning document. It does not implement updater changes, menu UI changes, README reduction, release prep, packaging, appcast publishing, version/build bumps, event-tap changes, or runtime behavior changes.

## Current Baseline

- Latest public release: `v1.0.0` / build `39`.
- Expected local `main` baseline for this planning branch: `711a76d810f993b8c1c8e57730fd57c26476988d`.
- Expected `origin/main` baseline for this planning branch: `711a76d810f993b8c1c8e57730fd57c26476988d`.
- The live Sparkle appcast is published and verified for `v1.0.0` / build `39`.
- Public update payloads remain GitHub Release DMG assets.
- The appcast remains hosted from GitHub Pages at `https://gergoterek.github.io/HoverClick/appcast.xml`.
- Event tap and mouse semantics are unchanged from v1.0.0: left and right mouse-down only, original events pass through unchanged, no synthetic clicks, no event replay, no cursor movement, no mouse-move focus, no scroll focus, and no delayed click delivery.
- App identity remains `HoverClick` / `com.gergoterek.HoverClick`.
- Sparkle private key material must remain outside the repository, release assets, appcast, logs, and documentation.

## v1.1.x Intent

The v1.1.x line should be a set of smaller incremental improvements after v1.0.0, not a risky rewrite. The likely first feature release is `v1.1.0`, with possible follow-up fixes or polish releases such as `v1.1.1`, `v1.1.2`, and `v1.1.3`.

Validation and release discipline remain strict:

- keep feature implementation separate from this planning branch;
- keep release prep separate from feature branches;
- keep appcast publication inside explicit release workflow only;
- stop if private key material, unexpected release assets, dirty runtime files, or source behavior changes appear outside the intended branch scope;
- preserve the narrow event tap and original-event pass-through contract unless a separate, explicit risky-event-semantics task is approved.

## Feature 1/3: Updater Completion

### Goals

- Make the update experience feel complete, clear, and reliable.
- Keep manual `Check for Updates...` understandable as the explicit user-initiated update action.
- Keep `Automatically Check for Updates` understandable as a persistent preference for periodic checks only.
- Make diagnostics clearly show updater state, including automatic checks, automatic download/install state, and whether automatic install is allowed.
- Keep the appcast workflow based on GitHub Release DMG assets plus the GitHub Pages appcast.
- Preserve private Sparkle key handling through the existing Keychain-based workflow.

### Non-Goals

- Do not enable silent or background automatic install unless explicitly approved in a later release decision.
- Do not change Sparkle private key storage or expose private key material.
- Do not replace Sparkle with a custom updater.
- Do not change app name, bundle identifier, signing identity, Sparkle public key, feed URL, or update-hosting strategy as incidental updater polish.
- Do not package a DMG, create a tag, create a GitHub Release, upload release assets, publish `gh-pages`, or modify `appcast.xml` in an updater implementation branch.

### Candidate Implementation Areas

- Menu wording and placement for `Check for Updates...` and `Automatically Check for Updates`.
- Diagnostics copy output for updater state, last update-check state if safely available, and automatic download/install policy.
- Sparkle controller or delegate hooks only where needed for clear state reporting or user feedback.
- Documentation that separates manual checks, automatic checks, and automatic install.

### Manual Test Expectations

Manual Finder UI validation -- not run automatically.

- `Check for Updates...` remains visible, understandable, and user-initiated.
- `Automatically Check for Updates` can be turned on and off and persists across relaunch.
- Enabling automatic checks does not enable automatic download or install.
- Diagnostics accurately report automatic checks and automatic download/install state.
- Update checking still reaches the live appcast without a 404.
- Update discovery from an older updater-enabled build to the new release is validated during release workflow, not during planning.

### Safety Constraints

- No silent/background automatic install without explicit approval.
- No private Sparkle key material in the repository, app bundle, appcast, GitHub Release assets, docs, or logs.
- Sparkle appcast workflow remains GitHub Release DMG plus `gh-pages` appcast.
- Appcast changes belong only to explicit release workflow.
- Event tap and mouse semantics remain unchanged.

## Feature 2/3: Menu UI/UX Refactor

### Goals

- Make the status menu feel calmer, more native, and easier to scan.
- Add clear section headers where native menu behavior supports them cleanly.
- Group related items instead of leaving a flat cluttered menu.
- Improve spacing and alignment.
- Avoid the messy `icon + space + label` look where a native menu image, checkmark, indentation level, or section structure would be clearer.
- Keep the menu compact enough for repeated daily use.

### Non-Goals

- Do not change click-focus behavior.
- Do not change event tap inputs or mouse semantics.
- Do not add hover focus, scroll focus, key focus, synthetic clicks, cursor movement, event replay, or delayed click delivery.
- Do not turn diagnostics into a persistent technical dashboard in the menu.
- Do not replace the native menu with fragile custom window/menu UI unless a later design explicitly proves that is necessary and safe.

### Candidate Menu Structure

Likely sections:

- Status / version
- General
- Click Focus
- Permissions
- Updates
- Diagnostics
- Quit

Candidate grouping:

- Status / version: `HoverClick`, visible short version, and concise status only if it stays calm.
- General: `About HoverClick...` if that feels more natural near the top.
- Click Focus: `Left Click Focus`, `Right Click Focus`.
- Permissions: Accessibility status, refresh/check action, `Launch at Login`, `Open Accessibility Settings`.
- Updates: `Check for Updates...`, `Automatically Check for Updates`.
- Diagnostics: `Verbose Diagnostics`, `Copy Diagnostics Summary`.
- Quit: `Quit`.

### Manual Test Expectations

Manual Finder UI validation -- not run automatically.

- Section grouping is clear and not visually noisy.
- Toggle states use native checkmarks.
- Action rows are aligned and do not show awkward inline `icon + space + label` text.
- Spacing feels native on current macOS.
- `Left Click Focus` and `Right Click Focus` remain enabled only when Accessibility allows them.
- `Check for Updates...`, `Automatically Check for Updates`, `Diagnostics`, `About HoverClick...`, and `Quit` remain discoverable.
- Cmd+Q still quits.

### Safety Constraints

- This is a UI/menu polish task, not an event-semantics task.
- The implementation must not change event tap mask, click pass-through behavior, Accessibility/TCC handling, bundle identity, signing, updater feed, appcast, or release assets.
- Any UI validation involving Finder or macOS menu interaction is manual only and must not be automated.

## Feature 3/3: README Reduction

### Goals

- Reduce the GitHub README to roughly one half or one third of its current length.
- Keep a concise product explanation that makes clear HoverClick is click-to-focus, not hover-to-focus.
- Keep install, update, permissions, troubleshooting, safety, and build entry points.
- Move technical depth into docs instead of deleting useful material.
- Make GitHub first impressions lighter while preserving enough detail for cautious users.

### Non-Goals

- Do not change product behavior, release workflow, build scripts, app metadata, updater settings, or appcast.
- Do not remove important safety promises from the project; move them to focused docs if they are too detailed for README.
- Do not reduce README in this planning branch.

### Candidate README Structure

- Short title and one-paragraph product summary.
- Compact feature list.
- Download and install.
- Permissions.
- Updates.
- Troubleshooting entry points.
- Privacy and safety summary.
- Build from source.
- Links to deeper docs.

### Docs Migration Strategy

- Move release-history detail to `docs/CURRENT_STATE.md`, `docs/RELEASE.md`, or a future release history doc.
- Move updater workflow depth to `docs/UPDATER_DESIGN.md` and `docs/APPCAST_RELEASE_WORKFLOW.md`.
- Move menu behavior and diagnostics details to `docs/ARCHITECTURE.md`, `docs/WORKFLOW.md`, and `docs/TEST_MATRIX.md`.
- Keep README as the public front door, not the full engineering notebook.

## Suggested Branch And Release Sequence

Recommended sequence:

1. `plan-v1.1-updater-menu-readme`
2. `feature-v1.1-updater-completion`
3. `feature-v1.1-menu-ui-refactor`
4. `feature-v1.1-readme-reduction`
5. `release-v1.1.0-prep`

Separate implementation branches are safer because the three items have different risk profiles:

- updater completion touches Sparkle/user-facing update state and needs careful diagnostics/release-workflow checks;
- menu UI/UX refactor touches native AppKit menu structure and needs manual UI review;
- README reduction is docs-only but can be reviewed better after the feature and menu wording settles.

A combined implementation branch should be considered only if the actual code changes turn out extremely small and tightly coupled. Even then, release prep should remain separate from implementation.
